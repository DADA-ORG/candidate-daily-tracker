# 候选人每日追踪工具 — 设计方案 v1

## 1. 背景与目标

顾问团队每天在 ATS 系统中导出候选人流程表（Excel），但表格分散、人工翻查耗时，容易漏掉需要跟进的候选人。本工具的目标：把多张导出表自动转成"每位顾问的每日待办清单"，以飞书私信卡片的形式直接推送给对应顾问，减少人工翻表和遗漏。

## 2. 整体流程

```
顾问上传 Excel 表（1~2 张）
        ↓
工具识别字段、按规则筛选出需要跟进的记录
        ↓
按"负责顾问"字段匹配到飞书用户（通过姓名对照表）
        ↓
为每位顾问生成一张汇总卡片
        ↓
通过飞书机器人私信发送给对应顾问
```

触发方式：**人工触发**。顾问把当天的表发给工具（本场景下即上传给 Claude），说一声"处理"即可，不做成定时自动任务。

## 3. 数据源说明

目前确认的数据来自 ATS 系统按"阶段（Process）"导出的两张表，表头结构一致：

| 阶段（Process 值） | 文件含义 | 关键专属字段 |
|---|---|---|
| CV Sent | 简历已发送、等待客户反馈 | Date Added（简历发送日期） |
| CCM | 客户已确认面试 | Client Interview Interview Date（面试时间） |

共有字段：Candidate（候选人姓名）、Client Name（客户）、Job（岗位）、**User（负责顾问，匹配飞书的关键字段）**、Last Update（最后更新时间）、Process（阶段标记）、Candidate Status（Active / Follow Up / Inactive / 空）、Candidate Tags（备注标签）、Date Added（记录创建日期）。

**关键发现**：同一候选人的 `Date Added` 和 `Last Update` 可能相差很多天（例如 Date Added 是 5-19，Last Update 却是 6-24）。这说明 `Date Added` 才是"简历实际发送日期"的正确锚点，不能用 Last Update 代替，否则天数会算错。

后续如果顾问只导出这两类表，字段映射可以直接按上表硬编码；如果未来出现新的 Process 阶段（如 Offer、Onboard），需要另行补充规则——目前先按这两种阶段设计，不做过度泛化。

## 4. 业务规则（核心逻辑）

### 规则 1：今日需跟进岗位
来源：CV Sent 表 + CCM 表中，按 `User`（顾问）分组，取该顾问名下所有 `(Client Name, Job)` 组合，**去重**后列出。
说明：不做日期判断，只要该岗位出现在当天上传的表里，就视为"今日在跟进中"。

### 规则 2：简历已发送 ≥ 3 天，需找客户要反馈
来源：CV Sent 表
条件：`今天 − Date Added ≥ 3 天`
不筛选 Candidate Status（Inactive 的候选人也会被提醒，因为可能只是标记有误或仍需跟进关闭）。

### 规则 3：已面试 ≥ 1 天，需找客户要反馈
来源：CCM 表
条件：`今天 − Client Interview Interview Date ≥ 1 天`
同样不筛选 Candidate Status。

### 重复提醒策略
不做"已提醒"记录。只要候选人状态没有变化（没有从表里移除或标记推进），每次运行都会重复提醒，直到该候选人不再满足条件为止。好处是实现简单、不会漏提醒；代价是连续几天跑会看到同一条提醒——这是产品有意的选择（宁可重复也不遗漏）。

## 5. 顾问匹配机制

顾问已在飞书搭建了一张多维表格（Bitable）作为姓名对照表：包含"姓名"文本字段 + "关联人员"字段（该字段本身就关联到真实飞书账号，可直接拿到 open_id，无需再做模糊文本匹配）。

工具运行时：拿 Excel 里的 `User`（顾问英文名）→ 查对照表的姓名字段 → 取关联人员字段对应的 open_id → 作为消息接收对象。

**待确认**：需要拿到这张 Bitable 的 App Token 和 Table ID（或者表格分享链接），以及"姓名"字段的准确列名，才能写读取逻辑。

## 6. 消息卡片设计

每位顾问收到一张卡片，按当天匹配到的记录分三个板块展示：

```
📋 [顾问名] 今日候选人跟进清单 — 2026-07-10

【今日跟进岗位】（去重）
· 客户A - 岗位A
· 客户B - 岗位B

【简历待反馈】（发送 ≥3 天）
· 候选人X - 岗位A - 已发送 5 天（发送日期 06-05）
· 候选人Y - 岗位B - 已发送 3 天（发送日期 07-07）

【面试待反馈】（面试 ≥1 天）
· 候选人Z - 岗位C - 面试日期 07-08，已过 2 天
```

如果某个板块当天没有匹配记录，直接省略该板块，避免空卡片刷屏。

## 7. 发送机制

通过飞书自建应用的机器人身份，调用消息发送接口，以 `open_id` 为接收对象发送单聊卡片消息（interactive 卡片类型）。

**前置条件（目前还未就绪）**：需要在飞书开放平台创建一个自建应用，并开通"获取与发送单聊、群组消息"权限，机器人可见范围需要覆盖所有顾问。这一步需要公司飞书管理员/IT 协助创建应用并拿到 App ID / App Secret。

## 8. 待办清单（下一步需要准备的东西）

1. 创建飞书自建应用，开通单聊消息发送权限，拿到 App ID / App Secret
2. 提供姓名对照表 Bitable 的分享链接（或 App Token + Table ID），确认字段名
3. 确认后即可进入实现阶段：脚本读取上传的 Excel → 按规则筛选 → 匹配顾问 → 生成卡片 → 调用飞书 API 发送

## 9. 后续可扩展方向（暂不实现）

- 改为定时任务（每天固定时间自动读取固定路径的表格并发送），无需人工触发
- 增加"已提醒记录"，避免连续多天重复提醒同一条
- 支持更多 Process 阶段（Offer、Onboard 等）的规则
- 群内汇总卡片 + 私信提醒并存
